## Strict Configuration & Setup Instructions

## 1. Bootstrap via the Official Wizard [7]

* Do not use old third-party templates or deprecated Moko-Resources setups.
* Generate your boilerplate directly using the official [Kotlin Multiplatform Wizard](https://kmp.jetbrains.com/). Select Android and iOS, and choose Share UI (Compose Multiplatform). [8, 9, 10, 11, 12]

## 2. Enforce the Latest Stable Toolchain [13]

In your root build.gradle.kts, always target the newest language capabilities and compiler tools to prevent standard build warnings.

* Use Kotlin 2.1.x+ (which utilizes the stable K2 compiler architecture).
* Target Java 17 or Java 21 across your compile choices. [14]

```kotlin
// Example strict compiler configuration in shared/build.gradle.kts
kotlin {
    jvmToolchain(17) // Enforces Java 17 minimum across environments
}
```

## 3. Use Strict Type-Safe Navigation [15]

Avoid string-based route definitions which are now deprecated. Use Kotlin @Serializable objects to define routes. [16, 17]

```kotlin
// Define your destinations as type-safe objects
@Serializable
object Home

@Serializable
data class Profile(val userId: String)

// Implement inside your shared Compose code
NavHost(navController = navController, startDestination = Home) {
    composable<Home> { HomeScreen(navController) }
    composable<Profile> { backStackEntry ->
        val profile: Profile = backStackEntry.toRoute()
        ProfileScreen(profile.userId)
    }
}
```

## 4. Handle iOS Native Lifecycle and Framework Packs

* Ensure your iosApp Xcode project links the shared logic using the modern Cocoapods-free embed framework script or standard Swift Package Manager (SPM) integration generated by the wizard.
* Keep iOS rendering smooth by utilizing ComposeUIViewController inside your Swift entry point: [18]

```swift
import SwiftUI
import shared // Your KMP shared module

struct ContentView: View {
    var body: some View {
        ComposeView().ignoresSafeArea(.all, edges: .bottom)
    }
}

struct ComposeView: UIViewControllerRepresentable {
    func makeUIViewController(context: Context) -> UIViewController {
        MainViewControllerKt.MainViewController() // Exported from Kotlin
    }
    func updateUIViewController(_ uiViewController: UIViewController, context: Context) {}
}
```

## ✅ Summary of Ready Stack

Your project will run entirely on Kotlin Multiplatform + Compose Multiplatform, using Koin (DI), Ktor (API networking), Room (Database), Coil3 (Images), and Type-Safe Compose Navigation.

To build 5+ distinct apps using a shared core stack while allowing each app to have its own independent version and release lifecycle, you must switch from a strict Monorepo to an Internal SDK / Remote Artifact Architecture. [1, 2]

Putting 5 apps with independent versions into one local Gradle monorepo breaks Gradle's build cache optimizations, causes dependency lock-ins, and forces you to compile apps you aren't testing. Isolating your shared stack into a versioned SDK solves this. [3]

## 1. Codebase Architecture: The Hub-and-Spoke Model

You will maintain 6 distinct Git repositories: 1 Core Stack Engine Repository, and 5 separate Application Repositories. [4, 5]

```
┌────────────────────────────────────────────────────────┐
│            1. CORE STACK REPOSITORY (:sdk)             │
│  (Shares Ktor, Room, Design Tokens, Auth, State Engine)│
└───────────────────────────┬────────────────────────────┘
                            │
          Publishes versioned artifacts (e.g., v1.4.2, v2.0.0-rc) via CI/CD
                            │
    ┌─────────────────────┼─────────────────────┐
    ▼                     ▼                     ▼
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  2. APP ONE  │      │  2. APP TWO  │      │ 2. APP THREE │
│ (Uses v1.4.2)│      │ (Uses v2.0.0)│      │ (Uses v1.4.2)│
└──────────────┘      └──────────────┘      └──────────────┘
```

## Repositories Layout

1. **The Core Engine Repo** (`/company-core-sdk`): This contains your pure Compose Multiplatform code, common data architectures (Ktor, Room), and foundational layout containers. It does not contain an executable app—only library modules. [3, 6, 7, 8]
2. **The App Repos** (`/app-ecommerce`, `/app-fitness`, etc.): Each app is a lean, completely independent project containing only its platform shells (androidApp, iosApp), custom asset assets, and specialized features. [9, 10]

## 2. The Shared Stack & Versioning Pipeline

To allow independent versioning without breaking apps, the Core SDK must publish pre-compiled multiplatform binaries via Maven Central or a private instance like GitHub Packages. [1]

### Continuous Delivery for the Stack

Every time you merge code into your Core SDK repo, your CI pipeline bumps the version automatically:

```yaml
# Core SDK Github Action Workflow Example
on:
  push:
    tags:
      - 'v*' # Triggers on v1.0.1, v1.1.0-alpha, etc.

jobs:
  publish:
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v4
      - name: Publish KMP Frameworks to GitHub Packages
        run: ./gradlew publishAllPublicationsToGitHubPackagesRepository
```

### Consuming the Engine in Apps

Each separate application project has its own libs.versions.toml. To upgrade or pin a specific version of your shared architecture, you change exactly one line inside that specific app's version catalog: [11]

```toml
# Inside /app-ecommerce/gradle/libs.versions.toml
[versions]
companyCoreSdk = "2.1.0" # App One can upgrade immediately

[libraries]
shared-ui = { module = "com.company.core:shared-ui", version.ref = "companyCoreSdk" }
shared-data = { module = "com.company.core:shared-data", version.ref = "companyCoreSdk" }
```

## 3. Strict Git Strategy & Isolation Rules

Because the apps are separated, you eliminate the risk of one developer accidentally breaking all five apps with a single commit. [12]

### Git Branching Rules

* **Core Repo:** Uses rigid Semantic Versioning (SemVer) (e.g., MAJOR.MINOR.PATCH).
* Breaking changes to the core stack (e.g., updating to a new version of Ktor or changing database structures) require a MAJOR version bump. [13, 14]
* **App Repos:** Maintain completely isolated Git branch flows. App One can safely branch, test, and deploy to production without ever pulling or pulling code from App Two. [15]

### The "Local Development" Override (Composite Builds)

The biggest pain of an multi-repo SDK structure is editing core code and wanting to see it instantly update in your app without waiting for an artifact to publish. [16]

To prevent developers from bypassing the system, use Gradle Composite Builds inside your apps to substitute the remote dependency with a local directory path when working locally: [17, 18]

```kotlin
// Inside /app-ecommerce/settings.gradle.kts
pluginManagement { ... }

// Conditional inclusion for seamless local debugging
if (extra.has("useLocalCore") && extra.get("useLocalCore") == "true") {
    includeBuild("../company-core-sdk") {
        dependencySubstitution {
            substitute(module("com.company.core:shared-ui")).using(project(":shared-ui"))
        }
    }
}
```

## 🚨 Final Architectural Instructions

* **Zero Local Project Dependencies:** Apps are strictly forbidden from directly linking to internal core module folders via local path scripts relative to their root directory.
* **Forced Dependency Lock:** Apps must consume the core stack as version-tagged remote Maven binaries. [1]
* **Isolated CI/CD Pipelines:** App deployment cycles run independently of each other. App One can release 4 times a week, while App Two is updated only once a month.

Avoid AI slops Ideas (like: gradients, emojis, over-reliance on "inter" fonts, Missing Accessibility: Lack of ARIA labels, low color contrast in dark modes, and poor HTML landmarks, uniform component sizing.)

Follow strictly the Modern Android Architecture (MVVM/MVI) with Clean Architecture.

**Avoid "One File Growing":** Strictly enforce logic separation. Break or distinct packages at minimum.

**Reusable Palettes:** Create a dedicated `designsystem` package for reusable Compose components (buttons, text styles, colors) to ensure UI consistency.

**Single Source of Truth:** Enforce a strict unidirectional data flow. Data layer models must be mapped to distinct UI models to prevent UI state logic from bleeding into or duplicating data processing logic.

**Pre-Refactoring Analysis:** Before writing any new function, class, or helper logic, scan the existing codebase to ensure similar logic does not already exist. Always prefer reusable palettes, aesthetics, items, files and folders to keep the codebase structured and readable.
